authorization\n\nVerified Code examples on this page have been automatically tested and verified.\n\nNote\n\nThis policy works only for \n\nMCP\n\nhttps://modelcontextprotocol.io/\n\n traffic. Note that all standard HTTP policies also apply to MCP traffic.\n\nThe MCP authorization policy works similarly to \n\nHTTP authorization\n\nhttps://agentgateway.dev/docs/standalone/latest/configuration/security/http-authz\n\n, but runs in the context of an MCP request. Authorization rules are written as \n\nCEL expressions\n\nhttps://agentgateway.dev/docs/standalone/latest/reference/cel\n\n that evaluate against specific MCP method invocations, such as \n\nlist_tools\n\n and \n\ncall_tools\n\n , to control which tools, prompts, and resources clients can access.\n\nIf a tool or other resource is not allowed, agentgateway automatically filters the resource from \n\nlist\n\n responses so that unauthorized clients never see it.\n\nThe \n\nmcpAuthorization\n\n policy can be attached at the backend level (applying to all MCP targets) or at the individual target level. For more on how policies inherit and override at different levels, see \n\nMCP target policies\n\nhttps://agentgateway.dev/docs/standalone/latest/mcp/mcp-target-policies\n\n.\n\nAllow specific tools\n\nThe following configuration exposes an MCP server and allows any client to call the \n\necho\n\n tool. All other tools from the server are blocked.\n\n# yaml-language-server: $schema=https://agentgateway.dev/schema/config\nbinds:\n- port: 3000\n  listeners:\n  - routes:\n    - backends:\n      - mcp:\n          targets:\n          - name: everything\n            stdio:\n              cmd: npx\n              args: [\"@modelcontextprotocol/server-everything\"]\n            policies:\n              mcpAuthorization:\n                rules:\n                - 'mcp.tool.name == \"echo\"'\n\n\nRole-based access with JWT claims\n\nWhen you combine MCP authorization with \n\nMCP authentication\n\nhttps://agentgateway.dev/docs/standalone/latest/mcp/mcp-authn\n\n, you can write rules that reference JWT claims. The following configuration restricts tools based on the authenticated user's identity and role.\n\nIn this example:\n\nThe MCP authentication policy validates JWTs against a local authorization server, such as Keycloak, running on port 9000.\n\nAny authenticated user can call the \n\necho\n\n tool.\n\nOnly the user \n\ntest-user\n\n can call the \n\nadd\n\n tool.\n\nOnly users with the nested claim \n\nnested.key == \"value\"\n\n can call the \n\nprintEnv\n\n tool.\n\n# yaml-language-server: $schema=https://agentgateway.dev/schema/config\nbinds:\n- port: 3000\n  listeners:\n  - routes:\n    - backends:\n      - mcp:\n          policies:\n            mcpAuthentication:\n              issuer: http://localhost:9000\n              jwks:\n                url: http://localhost:9000/.well-known/jwks.json\n              resourceMetadata:\n                resource: http://localhost:3000/mcp\n                scopesSupported:\n                - read:all\n                bearerMethodsSupported:\n                - header\n          targets:\n          - name: everything\n            stdio:\n              cmd: npx\n              args: [\"@modelcontextprotocol/server-everything\"]\n            policies:\n              mcpAuthorization:\n                rules:\n                # Any authenticated user can call 'echo'\n                - 'mcp.tool.name == \"echo\"'\n                # Only the test-user can call 'add'\n                - 'jwt.sub == \"test-user\" && mcp.tool.name == \"add\"'\n                # Claim-based access for 'printEnv'\n                - 'mcp.tool.name == \"printEnv\" && jwt.nested.key == \"value\"'\n\n\nDifferent rules per target\n\nWhen you multiplex several MCP servers behind a single agentgateway listener, you can apply different authorization rules to each target.\n\nIn this example:\n\nAny user can access public tools.\n\nOnly users with \n\nadmin\n\n in the JWT roles can access admin tools.\n\nThe JWT is validated against a local authorization server running on port 9000.\n\n# yaml-language-server: $schema=https://agentgateway.dev/schema/config\nbinds:\n- port: 3000\n  listeners:\n  - routes:\n    - policies:\n        cors:\n          allowOrigins: [\"*\"]\n          allowHeaders:\n          - mcp-protocol-version\n          - content-type\n          - cache-control\n      backends:\n      - mcp:\n          policies:\n            mcpAuthentication:\n              mode: optional\n              issuer: http://localhost:9000\n              jwks:\n                url: http://localhost:9000/.well-known/jwks.json\n              resourceMetadata:\n                resource: http://localhost:3000/mcp\n                scopesSupported:\n                - read:all\n                bearerMethodsSupported:\n                - header\n          targets:\n          - name: public-tools\n            stdio:\n              cmd: npx\n              args: [\"@modelcontextprotocol/server-everything\"]\n            policies:\n              mcpAuthorization:\n                rules:\n                # Allow anyone to access all tools\n                - 'true'\n\n          - name: admin-tools\n            stdio:\n              cmd: npx\n              args: [\"@mycompany/admin-server\"]\n            policies:\n              mcpAuthorization:\n                rules:\n                # Only authenticated admins can access these tools\n                - 'has(jwt.sub) && \"admin\" in jwt.roles'\n\n\nCEL expression reference\n\nAuthorization rules use \n\nCommon Expression Language (CEL)\n\nhttps://agentgateway.dev/docs/standalone/latest/reference/cel\n\n expressions. Rules are evaluated as an \n\nOR\n\n expression: if any rule matches, the request is allowed.\n\nReview the following table of common variables in MCP authorization rules.
