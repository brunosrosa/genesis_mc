expressions\n\nAgentgateway uses the CEL \n\nCEL (Common Expression Language)\n\n A simple expression language used throughout agentgateway to enable flexible configuration. CEL expressions can access request context, JWT claims, and other variables to make dynamic decisions. expression language throughout the project to enable flexibility. CEL allows writing simple expressions based on the request context that evaluate to some result.\n\nCEL playground\n\nYou can try out CEL expressions directly in the built-in CEL playground in the agentgateway admin UI. The playground uses agentgateway's actual CEL runtime, so custom functions and variables specific to agentgateway are available for testing.\n\nTo open the playground:\n\nRun agentgateway.\n\nagentgateway -f config.yaml\n\n\nOpen the \n\nCEL playground\n\nhttp://localhost:15000/ui/cel/\n\n.\n\nIn the \n\nExpression\n\n box, enter the CEL expression that you want to test.\n\nIn the \n\nInput Data (YAML)\n\n box, paste the YAML file structure that the CEL expression is running against.\n\nTo test your CEL expression, click \n\nEvaluate\n\n.\n\n \n\n \n\nExample expressions\n\nReview the following examples of expressions for different uses cases.\n\nDefault function\n\nYou can use the \n\ndefault\n\n function to provide a fallback value if the expression cannot be resolved.\n\nRequest header fallback with example of an anonymous user:\n\ndefault(request.headers[\"x-user-id\"], \"anonymous\")\n\n\nNested object fallback with example of light theme:\n\ndefault(json(request.body).user.preferences.theme, \"light\")\n\n\nJWT \n\nJWT (JSON Web Token)\n\n A compact, URL-safe token format used for securely transmitting information between parties. JWTs are commonly used for authentication and authorization in agentgateway. claim fallback with example of default “user” role:\n\ndefault(jwt.role, \"user\")\n\n\nLogs, traces, and observability\n\n# Include logs where there was no response or there was an error\nfilter: |\n   !has(response) || response.code > 299\nfields:\n  add:\n    user.agent: 'request.headers[\"user-agent\"]'\n    # A static string. Note the expression is a string, and it returns a string, hence the double quotes.\n    span.name: '\"openai.chat\"'\n    gen_ai.usage.prompt_tokens: 'llm.input_tokens'\n    # Parse the JSON request body, and conditionally log...\n    # * If `type` is sum, val1+val2\n    # * Else, val3\n    # Example JSON request: `{\"type\":\"sum\",\"val1\":2,\"val2\":3,\"val4\":\"hello\"}`\n    json.field: |\n      json(request.body).with(b,\n         b.type == \"sum\" ?\n           b.val1 + b.val2 :\n           b.val3\n      )\n\n\nAuthorization\n\nmcpAuthorization:\n  rules:\n  # Allow anyone to call 'echo'\n  - 'mcp.tool.name == \"echo\"'\n  # Only the test-user can call 'add'\n  - 'jwt.sub == \"test-user\" && mcp.tool.name == \"add\"'\n  # Any authenticated user with the claim `nested.key == value` can access 'printEnv'\n  - 'mcp.tool.name == \"printEnv\" && jwt.nested.key == \"value\"'\n\n\nRate limiting\n\nremoteRateLimit:\n   descriptors:\n   # Rate limit requests based on a header, whether the user is authenticated, and a static value (used to match a specific rate limit rule on the rate limit server)\n   - entries:\n      - key: some-static-value\n        value: '\"something\"'\n      - key: organization\n        value: 'request.headers[\"x-organization\"]'\n      - key: authenticated\n        value: 'has(jwt.sub)'\n\n\nCEL in YAML\n\nWhen writing CEL expressions in agentgateway, typically they are expressed as YAML string values, which can cause confusion. Any string literals within the expression need to be escaped with additional quotes.\n\nThese examples all represent the same CEL expression, a string literal \n\n\"hello\"\n\n .\n\ndoubleQuote: \"'hello'\"\ndoubleQuoteEscaped: \"\\\"hello\\\"\"\nsingleQuote: '\"hello\"'\nblockSingle: |\n  'hello'\nblockDouble: |\n  \"hello\"\n\n\nThese examples all represent the same CEL expression, a variable \n\nrequest\n\n :\n\ndoubleQuote: \"request\"\nsingleQuote: 'request'\nblock: |\n  request\n\n\nThe \n\nblock\n\n style is often the most readable for complex expressions, and also allows for multi-line expressions without needing to escape quotes.\n\nContext reference\n\nWhen using CEL expressions, a variety of variables and functions are made available.\n\nVariables\n\nVariables are only available when they exist in the current context. Previously in version 0.11 or earlier, variables like \n\njwt\n\n were always present but could be \n\nnull\n\n . Now, to check if a JWT claim exists, use the expression \n\nhas(jwt.sub)\n\n . This expression returns \n\nfalse\n\n if there is no JWT, rather than always returning \n\ntrue\n\n .\n\nAdditionally, fields are populated only if they are referenced in a CEL expression. This way, agentgateway avoids expensive buffering of request bodies if no CEL expression depends on the \n\nbody\n\n .\n\nEach policy execution consistently gets the current view of the request and response. For example, during logging, any manipulations from earlier policies (such as transformations or external processing) are observable in the CEL context.\n\nTable of variables\n\nField\n\nType\n\nDescription\n\nrequest\n\nobject\n\nrequest\n\n contains attributes about the incoming HTTP request\n\nrequest.method\n\nstring\n\nThe HTTP method of the request.
