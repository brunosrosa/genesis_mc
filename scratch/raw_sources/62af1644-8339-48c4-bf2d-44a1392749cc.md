is LLM Proxy?\n\nLLM Proxy is a tool that sits between your application and the different LLM providers. LLM Proxy's goal is to simplify the use of multiple LLMs through a TUI while providing cost and response optimization.\n\n( \n\nback to top\n\nhttps://github.com/llm-proxy/llm-proxy#readme-top\n\n)\n\nGetting Started\n\nThere are 2 ways to get started. You can directly clone the repo into your project or install it as a library.\n\nPrerequisites\n\nPython 3.11+\n\nInstallation\n\nWith pip:\n\npip install proxyllm\n\n\nWith poetry:\n\npoetry add proxyllm\n\n\nRun the install script for the default configuration file:\n\nconfig --default-config\n\n\nIf you prefer poetry:\n\npoetry run config --default-config\n\n\nIf the installation scripts do not work, you can visit the \n\nrepo\n\nhttps://github.com/llm-proxy/llm-proxy/blob/main/llmproxy.config.yml\n\n to grab a copy manually.\n\nNote:\n\nEnsure that you have all of your API keys for each respective provider in the .env file (You can utilize the .env.example for reference)\n\nFor Google's models, you will need the path to application credentials, and the project ID inside of the .env\n\n( \n\nback to top\n\nhttps://github.com/llm-proxy/llm-proxy#readme-top\n\n)\n\nUsage\n\nBasic Usage\n\nCurrently, the LLM Proxy provides 2 different route types: Cost and Category.\n\nTo get started import the LLMProxy client:\n\nfrom proxyllm import LLMProxy\n\n\nAfter the setup is complete, you only need 1 line of code to get started:\n\nllmproxy_client = LLMProxy()\n\n\nNote\n\n: You will need to specify your yaml configuration file if you did not use the default name:\n\nllmproxy_client = LLMProxy(path_to_user_configuration=\"llmproxy.config.yml\")\n\n\nTo use the llmproxy, simply call the route function with your prompt:\n\noutput = llmproxy_client.route(prompt=prompt)\n\n\nThe route function will return a CompletionResponse:\n\nprint(\"RESPONSE MODEL: \", output.response_model)\nprint(\"RESPONSE: \", output.response)\nprint(\"ERRORS: \", output.errors)\n\n\nresponse_model:\n\n contains the model used for the request\n\nresponse:\n\n contains the string response from the model\n\nerrors:\n\n contains an array of models that failed to make a request with their respective errors\n\nImportant Note:\n\n Although parameters changed programmatically, it is best to favor the YAML configuration file. Only use the constructor parameters when you must override the YAML configuration.\n\n( \n\nback to top\n\nhttps://github.com/llm-proxy/llm-proxy#readme-top\n\n)\n\nChat History\n\nThe user can pass in their own custom chat history to the proxy and it will route to the correct LLM with this chat history. The data type of the chat history is: \n\nchat_history: List[Dict[str,str]]\n\nThe chat history supports the following roles:\n\nsystem\n\n : Helps define the LLM's behavior. You can customize its personality or specify detailed instructions on how it should interact during conversations. It's important to note that while providing a system message is optional, if none is given, the assistant's behavior is likely to resemble that of a generic message, such as \"You are a helpful assistant.\" If you plan on using the system role, place it in the first index of \n\nchat_history\n\n rather than later on in the conversation.\n\nuser\n\n : Contain requests or comments for the LLM to address.\n\nassistant\n\n : Are the response messages by the LLMs, and can also be authored by you to showcase desired behaviors.\n\nAn example of how chat history could be formatted is as such:\n\nchat_history = [\n        {\n            \"role\": \"system\",\n            \"content\": \"you are math bot, and you're responses must be short and sweet\",\n        },\n        {\n            \"role\": \"user\",\n            \"content\": \"what is 1 + 1\",\n        },\n        {\n            \"role\": \"assistant\",\n            \"content\": \"2\",\n        }\n  ]\n\n\nBy default the chat history would be an empty array and will not need to be instantiated. But in order for you to retrieve the chat history after routing, simplyset your chat history variable to \n\noutput.chat_history\n\n and then this chat history can be passed into the proxy's route() function.\n\nThe following example shows the usage of chat history with the proxy library:\n\nprompt = \"what is 1 + 1\"\nproxy_client = LLMProxy(route_type=\"cost\")\noutput = proxy_client.route(prompt=prompt)\nchat_history = output.chat_history\n\nprompt2 = \"What was the first question that I asked you?\"\noutput = proxy_client.route(prompt=prompt2, chat_history=chat_history)\nchat_history = output.chat_history\n\n\nNotes:\n\nWhen passing in a populated custom chat history, the first message in the chat history has to be either a \n\nsystem\n\n or \n\nassistant\n\n message. The last message has to be an \n\nassistant\n\n message.
