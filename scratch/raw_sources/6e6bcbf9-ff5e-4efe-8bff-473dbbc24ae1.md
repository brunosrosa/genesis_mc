issue for preceding element\n\nAgentic LLM inference with long contexts is increasingly limited by memory bandwidth rather than compute. In this setting, SwiGLU MLP blocks, whose large weights exceed cache capacity, become a major yet under-optimized bottleneck. We propose DeepFusionKernel, a deeply fused kernel that cuts HBM traffic and boosts cache reuse, delivering up to 13.2% speedup on H100 and 9.7% on A100 over SGLang. Integrated with SGLang and paired with a kernel scheduler, DeepFusionKernel ensures consistent accelerations over generation lengths, while remaining adaptable to diverse models, inference configurations, and hardware platforms.\n\nReport issue for preceding element\n\nDeep Kernel Fusion for Transformers\n\nReport issue for preceding element\n\nZixi Zhang Imperial College London London, UK b.zhang25@imperial.ac.uk Zhiwen Mo Imperial College London London, UK zhiwen.mo25@imperial.ac.uk\n\nReport issue for preceding element\n\nYiren Zhao Imperial College London London, UK a.zhao@imperial.ac.uk Robert Mullins University of Cambridge Cambridge, UK robert.mullins@cl.cam.ac.uk\n\nReport issue for preceding element\n\n1 Introduction\n\nReport issue for preceding element\n\nThe Transformer architecture (Vaswani et al., \n\n2023\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib1\n\n) underpins modern large language models (LLMs) (Brown et al., \n\n2020\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib8\n\n; OpenAI et al., \n\n2024\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib10\n\n; DeepSeek-AI et al., \n\n2025a\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib7\n\n) , and as these models are deployed in agentic, real-world workloads, inference efficiency becomes a first-order concern (Aminabadi et al., \n\n2022\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib18\n\n; Zhou et al., \n\n2024\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib58\n\n; Yao et al., \n\n2023\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib61\n\n) . Agentic workloads require the model to handle very long contexts, maintain many persistent state items, and often produce long-form outputs (for example, large codebases). As a result, they process on the order of 100 × 100\\times more tokens per inference than typical chatbot-style workloads (Wu et al., \n\n2025\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib62\n\n) . In this regime, caching and fast access to KV values shift the primary bottleneck from raw compute to memory capacity and memory bandwidth. That shift in turn changes the optimization landscape: permissible batch sizes shrink, GEMM inputs become “fat,” Tensor Core utilization drops, and available compute is severely underused.\n\nReport issue for preceding element\n\nMost recent GPU optimizations focus on attention (Shah et al., \n\n2024\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib48\n\n; Kwon et al., \n\n2023\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib43\n\n; Zheng et al., \n\n2024\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib42\n\n; Ye et al., \n\n2025\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib50\n\n) , but in autoregressive decoding, the SwiGLU MLP blocks (Shazeer, \n\n2020\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib33\n\n; Grattafiori et al., \n\n2024\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib4\n\n; Bai et al., \n\n2023\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib5\n\n; DeepSeek-AI et al., \n\n2025b\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib6\n\n) dominate parameter count and drive memory-bandwidth pressure. Although modern accelerators offer high HBM bandwidth (NVIDIA Corporation, \n\n2021\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib37\n\n, \n\n2024\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib38\n\n, \n\n2025a\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib36\n\n) , memory scaling has not kept pace with compute, so memory-bound kernels remain the limiting factor for throughput on many real deployments.\n\nReport issue for preceding element\n\nTo close this gap, we propose aggressive deep kernel fusion to reduce memory traffic and improve utilization in FFN blocks in the attention mechanism. We implement a highly optimized fused operator, the DeepFusionKernel, which combines the separate GEMMs and pointwise kernels used in common four-kernel and two-kernel SwiGLU implementations (e.g., PyTorch (Paszke et al., \n\n2019\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib41\n\n) , SGLang (Zheng et al., \n\n2024\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib42\n\n) , and vLLM (Kwon et al., \n\n2023\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib43\n\n) ) into a single fused kernel that minimizes intermediate memory reads/writes and exposes more efficient work for Tensor Cores. Integrated with a lightweight, profile-driven kernel scheduler and deployed inside SGLang, DeepFusionKernel yields consistent, deployable speedups in bandwidth-bound agentic scenarios: up to 9.7% on A100 and 13.2% on H100 clusters compared to the SOTA SGLang implementation.\n\nReport issue for preceding element\n\n \n\nFigure 1: DeepFusionKernel leverages aggressive kernel fusion on SwiGLU blocks to eliminate intermediate activations and reduce memory traffic–without increasing FLOPs. Report issue for preceding element\n\n2 Background\n\nReport issue for preceding element\n\nModern Transformer-based LLMs replace the original two-layer ReLU design in the MLP with the gating-integrated SwiGLU variant (Shazeer, \n\n2020\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib33\n\n) :\n\nReport issue for preceding element\n\nA gate = X  W Gate , A 1 = X  W Up , A silu = SiLU  ( A gate ) = Sigmoid  ( A gate ) ⊗ A gate , A 2 = A 1 ⊗ A silu , Y = A 2  W Down . \\begin{aligned} &A_{\\rm gate}=XW_{\\rm Gate},\\quad A_{1}=XW_{\\rm Up},\\ &A_{\\rm silu}=\\mathrm{SiLU}(A_{\\rm gate})=\\mathrm{Sigmoid}(A_{\\rm gate})\\otimes A_{\\rm gate},\\ &A_{2}=A_{1}\\otimes A_{\\rm silu},\\quad Y=A_{2}W_{\\rm Down}.\\end{aligned}\n\nReport issue for preceding element\n\nHere W Up , W Gate ∈ ℝ d model × d ff W_{\\mathrm{Up}},W_{\\mathrm{Gate}}\\in\\mathbb{R}^{d_{\\mathrm{model}}\\times d_{\\mathrm{ff}}} and W Down ∈ ℝ d ff × d model W_{\\mathrm{Down}}\\in\\mathbb{R}^{d_{\\mathrm{ff}}\\times d_{\\mathrm{model}}} , with d ff d_{\\mathrm{ff}} being 3.5 to 4 times the d model d_{\\mathrm{model}} . These large matrices dominate the model's parameter count and memory footprint.\n\nReport issue for preceding element\n\n3 Method\n\nReport issue for preceding element\n\nKernel fusion (Aminabadi et al., \n\n2022\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib18\n\n; NVIDIA Corporation, \n\n2025b\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib46\n\n; NVIDIA, \n\n2018\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib17\n\n) reduces end-to-end latency and memory traffic by combining sequences of operations (GEMMs, pointwise nonlinearities, and simple elementwise math) into a single GPU kernel, thereby eliminating intermediate reads and writes–the dominant cost in bandwidth-bound LLM decoding (Kwon et al., \n\n2023\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib43\n\n; Zheng et al., \n\n2024\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib42\n\n) . Fusion is most effective for elementwise and tile-local computations; true reductions (e.g., Softmax) introduce long-range dependencies that limit cross-SM streaming and therefore are not good fusion targets. Guided by this observation, we split the Transformer MLP into two stages:\n\nReport issue for preceding element\n\nA 2 = ( X  W Up ) ⊗ SiLU  ( X  W Gate ) , Y = A 2  W Down . A_{2}=(XW_{\\mathrm{Up}})\\otimes\\mathrm{SiLU}(XW_{\\mathrm{Gate}}),\\quad Y=A_{2}W_{\\mathrm{Down}}.\n\nReport issue for preceding element\n\nand focus fusion effort on the first stage, which contains the GEMMs and pointwise gating that dominate memory traffic during autoregressive decoding.\n\nReport issue for preceding element\n\nAs illustrated in \n\nFigure ˜ 1\n\nhttps://arxiv.org/html/2602.11808v1#S1.F1\n\n, DeepFusionKernel fuses the separate GEMMs and pointwise kernels (four launches in a naive PyTorch implementation, two in SGLang/vLLM) into a single deeply fused operator. The fused kernel streams intermediate values through the computation, avoiding materialization of large temporaries and drastically cutting reads/writes to HBM. To realize this in practice, we systematically explore loop ordering and tiling strategies that maximize on-chip reuse and arithmetic intensity. Concretely, we observe the following trade-offs in our experiments:\n\nReport issue for preceding element\n\n• Row-major tiling: improves locality for the input activation X X , reducing repeated loads of input rows and benefiting scenarios with larger batch sizes or when activations dominate memory traffic. Report issue for preceding element\n\n• Column-major tiling: better reuses weight tiles, which is preferable when model weights dominate memory footprint (a common case in agentic decoding with small batch sizes and large models). Report issue for preceding element\n\nWe integrate these tiling and loop-ordering choices into the fused kernel implementation and tune tile sizes to balance register usage, shared-memory occupancy, and Tensor Core utilization. Because the best kernel configuration depends on model shapes, batch size, GPU microarchitecture, and distributed interconnects, we accompany DeepFusionKernel with a lightweight profiler-driven scheduler. At deployment time, the scheduler quickly benchmarks the set of candidate kernels prior to the inference on the target hardware and selects the highest-throughput option for the given model and workload, ensuring robust performance across architectures and real-world inference conditions.\n\nReport issue for preceding element\n\n4 Experiments\n\nReport issue for preceding element\n\nWe integrate DeepFusionKernel into a complete inference pipeline and measure end-to-end decoding throughput in a realistic framework to isolate practical benefits and deployment considerations. DeepFusionKernel with a profiler-driven scheduler consistently improves decoding throughput across realistic, bandwidth-bound inference scenarios. Key observations are:\n\nReport issue for preceding element\n\n• Robust gains in memory-bound regimes. Fusion produces the largest improvements when workloads are memory-bandwidth-limited (small batches, large models, long contexts). Even on H100, where compute is abundant, reducing memory traffic yields meaningful speedups. Report issue for preceding element\n\n• Stable behavior for long-context generation. As sequence length and KV cache grow, attention cost increases, but the MLP still contributes substantially to per-token cost; fusion therefore remains beneficial across long-generation workloads. Report issue for preceding element\n\n• Low deployment overhead. Report issue for preceding element\n\nOverall, DeepFusionKernel provides a practical, deployable improvement to existing inference stacks: it reduces memory traffic in the critical SwiGLU path, integrates with current frameworks, and produces repeatable throughput gains across hardware and workload regimes.\n\nReport issue for preceding element\n\n5 Related work\n\nReport issue for preceding element\n\nExisting frameworks like Apex (NVIDIA, \n\n2018\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib17\n\n) , TensorRT-LLM (NVIDIA Corporation, \n\n2025b\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib46\n\n) , and DeepSpeed-MII (Microsoft, \n\n2022\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib47\n\n) use shallow fusion (e.g., GEMM+activation), leaving significant overheads for larger MLPs. Automatic compile-time fusion approaches exist: Welder (Shi et al., \n\n2023\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib56\n\n) fuses based on tile-graph cost models but is limited to linear chains; TVM (Chen et al., \n\n2018\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib14\n\n) applies pattern matching and heuristics but its template-driven method mostly handles small trees; Blockbuster (Dekel, \n\n2025\n\nhttps://arxiv.org/html/2602.11808v1#bib.bib57\n\n) uses algebraic rules and demonstrated a SwiGLU prototype, but remains a standalone compiler study and lacks runtime feedback and hardware-aware tuning. In contrast, DeepFusionKernel deeply fuses the full SwiGLU into a single kernel. Paired with the kernel scheduler, it adapts fusion depth to workload and GPU, delivering consistent, branch-free speedups.\n\nReport issue for preceding element\n\n6 Conclusion\n\nReport issue for preceding element\n\nWe present DeepFusionKernel, an aggressively fused CUDA operator that eliminates intermediate buffers in the SwiGLU MLP and rebalances the trade-off between memory traffic and on-chip compute. When integrated into SGLang and driven by a lightweight profiler-based scheduler, the fused kernel produces consistent, deployable throughput improvements–up to 9.7% on A100 and 13.2% on H100–across batch sizes and long-generation agentic workloads. By targeting the memory-bandwidth bottleneck that dominates autoregressive decoding, DeepFusionKernel lets modern GPUs better realize their available compute, making it a practical optimization for real-world LLM inference pipelines.\n\nReport issue for preceding element\n\nLimitations\n\nReport issue for preceding element\n\nDue to limited computing resources, we do not exhaustively evaluate different GPU cluster interconnects, though our tests indicate that any resulting performance degradation is minimal for the workloads studied.
