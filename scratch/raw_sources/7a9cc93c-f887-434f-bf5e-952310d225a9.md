Why Skills\n\nhttps://codelabs.developers.google.com/getting-started-with-antigravity-skills#1\n\nModern AI agents have evolved from simple listeners to complex reasoners that integrate with local file systems and external tools (via \n\nMCP servers\n\nhttps://github.com/modelcontextprotocol/servers\n\n). However, indiscriminately loading an agent with entire codebases and hundreds of tools leads to \n\nContext Saturation\n\n and \"Tool Bloat.\" Even with large context windows, dumping 40–50k tokens of unused tools into active memory causes high latency, financial waste, and \"context rot,\" where the model becomes confused by irrelevant data.\n\nThe Solution: Agent Skills\n\nTo solve this, Anthropic \n\nintroduced Agent Skills\n\nhttps://claude.com/blog/equipping-agents-for-the-real-world-with-agent-skills\n\n, shifting the architecture from monolithic context loading to \n\nProgressive Disclosure\n\n. Instead of forcing the model to \"memorize\" every specific workflow (like database migrations or security audits) at the start of a session, these capabilities are packaged into modular, discoverable units.\n\nHow It Works\n\nThe model is initially exposed only to a lightweight \"menu\" of metadata. It loads the heavy procedural knowledge (instructions and scripts) only when the user's intent specifically matches a skill. This ensures that a developer asking to refactor authentication middleware gets security context without loading unrelated CSS pipelines, keeping the context lean, fast, and cost-effective.\n\n \n\n3. Agent Skills and Antigravity\n\nhttps://codelabs.developers.google.com/getting-started-with-antigravity-skills#2\n\nIn the Antigravity ecosystem, if the Agent Manager is the brain and the Editor is the canvas, \n\nSkills\n\n act as specialized training modules that bridge the gap between the generalist Gemini 3 model and your specific context. They allow the agent to \"equip\" a defined set of instructions and protocols—such as database migration standards or security checks—only when a relevant task is requested. By dynamically loading these execution protocols, Skills effectively transform the AI from a generic programmer into a specialist that rigorously adheres to an organization's codified best practices and safety standards.\n\nWhat is a Skill in Antigravity?\n\nIn the context of Google Antigravity, a Skill is a directory-based package containing a definition file ( \n\nSKILL.md\n\n ) and optional supporting assets (scripts, references, templates).\n\nIt is a mechanism for on-demand capability extension.\n\nOn-Demand\n\n: Unlike a System Prompt (which is always loaded), a Skill is only loaded into the agent's context when the agent determines it is relevant to the user's current request. This optimizes the context window and prevents the agent from being distracted by irrelevant instructions. In large projects with dozens of tools, this selective loading is crucial for performance and reasoning accuracy.\n\nCapability Extension\n\n: Skills can do more than just instruct; they can execute. By bundling Python or Bash scripts, a Skill can give the agent the ability to perform complex, multi-step actions on the local machine or external networks without the user needing to manually run commands. This transforms the agent from a text generator into a tool user.\n\nSkills v/s the Ecosystem (Tools, Rules and Workflows)\n\nWhile Model Context Protocol (MCP) functions as the agent's \"hands\"—providing heavy-duty, persistent connections to external systems like GitHub or PostgreSQL—Skills act as the \"brains\" that direct them.\n\nMCP handles the stateful infrastructure, whereas Skills are lightweight, ephemeral task definitions that package the methodology for using those tools. This serverless approach allows agents to execute ad-hoc tasks (like generating changelogs or migrations) without the operational overhead of running persistent processes, loading the context only when the task is active and releasing it immediately after.\n\nFunctionally, Skills occupy a unique middle ground between \"Rules\" (passive, always-on guardrails) and \"Workflows\" (active, user-triggered macros). Unlike workflows that require specific commands (e.g., \n\n/test\n\n ), Skills are \n\nagent-triggered\n\n: the model automatically detects the user's intent and dynamically equips the specific expertise required. This architecture allows for powerful composability; for example, a global Rule can enforce the use of a \"Safe-Migration\" Skill during database changes, or a single Workflow can orchestrate multiple Skills to build a robust deployment pipeline.\n\n4. Creating Skills\n\nhttps://codelabs.developers.google.com/getting-started-with-antigravity-skills#3\n\nCreating a Skill in Antigravity follows a specific directory structure and file format. This standardization ensures that skills are portable and that the agent can reliably parse and execute them. The design is intentionally simple, relying on widely understood formats like Markdown and YAML, lowering the barrier to entry for developers wishing to extend their IDE's capabilities.\n\nDirectory Structure\n\nSkills can be defined at two scopes, allowing for both project-specific and user-specific customizations :\n\nWorkspace Scope\n\n: Located in \n\n<workspace-root>/.agent/skills/\n\n . These skills are available only within the specific project. This is ideal for project-specific scripts, such as deployment to a specific environment, database management for that app, or generating boilerplate code for a proprietary framework.\n\nGlobal Scope\n\n: Located in \n\n~/.gemini/antigravity/skills/\n\n . These skills are available across all projects on the user's machine. This is suitable for general utilities like \"Format JSON,\" \"Generate UUIDs,\" \"Review Code Style,\" or integration with personal productivity tools.\n\nA typical Skill directory looks like this:\n\nmy-skill/\n├── SKILL.md # The definition file\n├── scripts/ # [Optional] Python, Bash, or Node scripts\n     ├── run.py\n     └── util.sh\n├── references/ # [Optional] Documentation or templates\n     └── api-docs.md\n└── assets/ # [Optional] Static assets (images, logos)\n\n\nThis structure separates concerns effectively. The logic ( \n\nscripts\n\n ) is separated from the instruction ( \n\nSKILL.md\n\n ) and the knowledge ( \n\nreferences\n\n ), mirroring standard software engineering practices.\n\nThe SKILL.md Definition File\n\nThe SKILL.md file is the brain of the Skill. It tells the agent what the skill is, when to use it, and how to execute it.\n\nIt consists of two parts:\n\nYAML Frontmatter\n\nMarkdown Body.\n\nYAML Frontmatter\n\nThis is the metadata layer. It is the only part of the skill that is indexed by the agent's high-level router. When a user sends a prompt, the agent semantic-matches the prompt against the description fields of all available skills.\n\n---\nname: database-inspector\ndescription: Use this skill when the user asks to query the database, check table schemas, or inspect user data in the local PostgreSQL instance.\n---\n\n\nKey Fields:\n\nname\n\n: This is not mandatory. Must be unique within the scope. Lowercase, hyphens allowed (e.g., \n\npostgres-query\n\n , \n\npr-reviewer\n\n ). If it's not provided, it will default to the directory name.\n\ndescription\n\n: This is mandatory and the most important field. It functions as the \"trigger phrase.\" It must be descriptive enough for the LLM to recognize semantic relevance. A vague description like \"Database tools\" is insufficient. A precise description like \"Executes read-only SQL queries against the local PostgreSQL database to retrieve user or transaction data. Use this for debugging data states\" ensures the skill is picked up correctly.\n\nThe Markdown Body\n\nThe body contains the instructions. This is \"prompt engineering\" persisted to a file. When the skill is activated, this content is injected into the agent's context window.\n\nThe body should include:\n\nGoal: A clear statement of what the skill achieves.\n\nInstructions: Step-by-step logic.\n\nExamples: Few-shot examples of inputs and outputs to guide the model's performance.\n\nConstraints: \"Do not\" rules (e.g., \"Do not run DELETE queries\").\n\nExample SKILL.md Body:\n\nDatabase Inspector\nGoal\nTo safely query the local database and provide insights on the current data state.\nInstructions\n- Analyze the user's natural language request to understand the data need.\n- Formulate a valid SQL query.\n - CRITICAL: Only SELECT statements are allowed.\n- Use the script scripts/query_runner.py to execute the SQL.\n - Command: python scripts/query_runner.py \"SELECT * FROM...\"\n- Present the results in a Markdown table.\nConstraints\n- Never output raw user passwords or API keys.\n- If the query returns > 50 rows, summarize the data instead of listing it all.\n\n\nScript Integration\n\nOne of the most powerful features of Skills is the ability to delegate execution to scripts. This allows the agent to perform actions that are difficult or impossible for an LLM to do directly (like binary execution, complex mathematical calculation, or interacting with legacy systems).\n\nScripts are placed in the \n\nscripts/\n\n subdirectory. The \n\nSKILL.md\n\n references them by relative path.\n\n5. Authoring Skills\n\nhttps://codelabs.developers.google.com/getting-started-with-antigravity-skills#4\n\nThe goal of this section is to build out Skills that integrate into Antigravity and progressively show various features like resources / scripts / etc.\n\nYou can download the Skills from the Github repo here: \n\nhttps://github.com/rominirani/antigravity-skills\n\nhttps://github.com/rominirani/antigravity-skills\n\n.\n\nWe can look at placing each of these skills inside of either \n\n~/.gemini/antigravity/skills\n\n folder or \n\n/.agent/skills\n\n folder.\n\nLevel 1 : The Basic Router (\n\n \n\ngit-commit-formatter\n\n \n\n)\n\nLet's consider this as the \"Hello World\" of Skills.\n\nDevelopers often write lazy commit messages e.g. \"wip\", \"fix bug\", \"updates\". Enforcing \"Conventional Commits\" manually is tedious and often forgotten. Let's implement a Skill that enforces the Conventional Commits specification. By simply instructing the agent on the rules, we allow it to act as the enforcer.\n\ngit-commit-formatter/\n└── SKILL.md  (Instructions only)\n\n\nThe \n\nSKILL.md\n\n file is shown below:\n\n---\nname: git-commit-formatter\ndescription: Formats git commit messages according to Conventional Commits specification. Use this when the user asks to commit changes or write a commit message.\n---\nGit Commit Formatter Skill\nWhen writing a git commit message, you MUST follow the Conventional Commits specification.\nFormat\n`<type>[optional scope]: <description>`\nAllowed Types\n- **feat**: A new feature\n- **fix**: A bug fix\n- **docs**: Documentation only changes\n- **style**: Changes that do not affect the meaning of the code (white-space, formatting, etc)\n- **refactor**: A code change that neither fixes a bug nor adds a feature\n- **perf**: A code change that improves performance\n- **test**: Adding missing tests or correcting existing tests\n- **chore**: Changes to the build process or auxiliary tools and libraries such as documentation generation\nInstructions\n1. Analyze the changes to determine the primary `type`.\n2. Identify the `scope` if applicable (e.g., specific component or file).\n3. Write a concise `description` in an imperative mood (e.g., \"add feature\" not \"added feature\").\n4. If there are breaking changes, add a footer starting with `BREAKING CHANGE:`.\nExample\n`feat(auth): implement login with google`\n\n\nHow to Run This Example:\n\nMake a small change to any file in your workspace.\n\nOpen the chat and type: Commit these changes.\n\nThe Agent will not just run git commit. It will first activate the git-commit-formatter skill.\n\nResult: A conventional Git commit message will be proposed.\n\nFor e.g. I made Antigravity add some comments to a sample Python file and it ended up with a Git commit message like \n\ndocs: add detailed comments to demo_primes.py.\n\nLevel 2: Asset Utilization (license-header-adder)\n\nThis is the \"Reference\" pattern.\n\nEvery source file in a corporate project might need a specific 20-line Apache 2.0 license header. Putting this static text directly into the prompt (or \n\nSKILL.md\n\n ) is wasteful. It consumes tokens every time the skill is indexed, and the model might \"hallucinate\" typos in legal text.\n\nOffloading the static text to a plain text file in a \n\nresources/\n\n folder. The skill instructs the agent to read this file only when needed.\n\nlicense-header-adder/\n├── SKILL.md\n└── resources/\n   └── HEADER_TEMPLATE.txt  (The heavy text)\n\n\nThe \n\nSKILL.md\n\n file is shown below:\n\n---\nname: license-header-adder\ndescription: Adds the standard open-source license header to new source files. Table names must be snake_case.\n   2. Every table must have a primary key named 'id'.\n   3. No 'DROP TABLE' statements allowed (safety).\n   \"\"\"\n   try:\n       with open(filename, 'r') as f:\n           content = f.read()\n          \n       lines = content.split('\\n')\n       errors = []\n      \n       # Check 1: No DROP TABLE\n       if re.search(r'DROP TABLE', content, re.IGNORECASE):\n           errors.append(\"ERROR: 'DROP TABLE' statements are forbidden.\")\n          \n       # Check 2 & 3: CREATE TABLE checks\n       table_defs = re.finditer(r'CREATE TABLE\\s+(?P<name>\\w+)\\s*\\((?P<body>.*?)\\);', content, re.DOTALL | re.IGNORECASE)\n      \n       for match in table_defs:\n           table_name = match.group('name')\n           body = match.group('body')\n          \n           # Snake case check\n           if not re.match(r'^[a-z][a-z0-9_]*$', table_name):\n               errors.append(f\"ERROR: Table '{table_name}' must be snake_case.\")\n              \n           # Primary key check\n           if not re.search(r'\\bid\\b.*PRIMARY KEY', body, re.IGNORECASE):\n               errors.append(f\"ERROR: Table '{table_name}' is missing a primary key named 'id'.\")\n       if errors:\n           for err in errors:\n               print(err)\n           sys.exit(1)\n       else:\n           print(\"Schema validation passed.\")\n           sys.exit(0)\n          \n   except FileNotFoundError:\n       print(f\"Error: File '{filename}' not found.\")\n       sys.exit(1)\nif __name__ == \"__main__\":\n   if len(sys.argv) != 2:\n       print(\"Usage: python validate_schema.py <schema_file>\")\n       sys.exit(1)\n      \n   validate_schema(sys.argv[1])\n\n\nHow to Run This Example:\n\nCreate a bad SQL file \n\nbad_schema.sql\n\n : \n\nCREATE TABLE users (name TEXT);\n\nType: \n\nValidate bad_schema.sql\n\n .\n\nThe agent does not guess.
