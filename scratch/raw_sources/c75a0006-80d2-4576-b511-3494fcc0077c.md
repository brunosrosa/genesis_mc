(Standalone)\n\nhttps://agentgateway.dev/docs/standalone/latest/\n\n / \n\nReference\n\nhttps://agentgateway.dev/docs/standalone/latest/reference/\n\n / Release notes\n\nRelease notes\n\nReview the release notes for agentgateway standalone.\n\nℹ\n\nFor more details, review the \n\nGitHub release notes in the agentgateway repository\n\nhttps://github.com/agentgateway/agentgateway/releases\n\n🔥 Breaking changes\n\nNew release version pattern\n\nThe previous release version pattern was changed to align the version number pattern for agentgateway standalone and agentgateway on Kubernetes. Going forward, both projects use the same release version number. If you have existing CI/CD workflows that depend on the old pattern, update them.\n\nCEL 2.0\n\nThis release includes a major refactor to the CEL implementation in agentgateway that brings substantial performance improvements and enhanced functionality. Individual CEL expressions are now 5-500x faster, which has improved end-to-end proxy performance by 50%+ in some tests. For more details on the performance improvements, see this \n\nblog post on CEL optimization\n\nhttps://blog.howardjohn.info/posts/cel-fast/\n\n.\n\nThe following user-facing changes were introduced:\n\nFunction name changes\n\n: For compatibility with the CEL-Go implementation, the \n\nbase64Encode\n\n and \n\nbase64Decode\n\n functions now use dot notation: \n\nbase64.encode\n\n and \n\nbase64.decode\n\n . The old camel case names remain in place for backwards compatibility.\n\nNew string functions\n\n: The following string manipulation functions were added to the CEL library: \n\nstartsWith\n\n , \n\nendsWith\n\n , \n\nstripPrefix\n\n , and \n\nstripSuffix\n\n . These functions align with the Google \n\nCEL-Go strings extension\n\nhttps://pkg.go.dev/github.com/google/cel-go/ext#Strings\n\n.\n\nNull values fail\n\n: If a top-level variable returns a null value, the CEL expression now fails. Previously, null values always returned true. For example, the \n\nhas(jwt)\n\n expression was previously successful if the JWT was missing or could not be found. Now, this expression fails.\n\nLogical operators\n\n: Logical \n\n||\n\n and \n\n&&\n\n operators now handle evaluation errors gracefully instead of propagating them. For example, \n\na || b\n\n returns \n\ntrue\n\n if \n\na\n\n is true even if \n\nb\n\n errors. Previously, the CEL expression failed.\n\nMake sure to update and verify any existing CEL expressions that you use in your environment.\n\nFor more information, see the \n\nCEL expression\n\nhttps://agentgateway.dev/docs/standalone/latest/reference/cel/\n\n reference.\n\nExternal auth fail-closed\n\nExternal auth policies now fail closed by default when the auth server is unreachable. This means requests are denied if the external authorization service cannot be reached. You are affected if you have an \n\nextAuthz\n\n policy configured and the auth service becomes unavailable.\n\nBefore\n\n: If the external auth service was unreachable, the behavior was undefined and requests could pass through.\n\nAfter\n\n: Requests protected by an external auth policy are rejected with a failure response until the auth service is reachable.\n\nTo explicitly allow requests when the auth service is unavailable, set the \n\nfailureMode\n\n to \n\nallow\n\n :\n\nextAuthz:\n  host: localhost:9000\n  failureMode: allow\n  protocol:\n    grpc: {}\n\n\nFor more information, see \n\nExternal authorization\n\nhttps://agentgateway.dev/docs/standalone/latest/configuration/security/external-authz/\n\n.\n\nMCP deny-only authorization policies\n\nA critical correctness bug was fixed in MCP authorization. You are affected if you have an MCP authorization policy that uses deny rules without any corresponding allow rules.\n\nFor example, review the following configuration. Previously, this policy denied all tool access, not just access to the \n\necho\n\n tool. Starting in 1.0.0, only \n\necho\n\n is denied and all other tools are allowed.\n\nmcpAuthorization:\n  rules:\n  - deny: 'mcp.tool.name == \"echo\"'\n\n\nMCP authentication mode change\n\nThe default MCP authentication mode now defaults to \n\nstrict\n\n mode instead of \n\npermissive\n\n . Requests to MCP backends without valid credentials are rejected by default. To restore the \n\npermissive\n\n behavior, set the \n\nmode\n\n field in your MCP authentication configuration:\n\nmcpAuthentication:\n  mode: permissive\n  issuer: http://localhost:9000\n  jwks:\n    url: http://localhost:9000/.well-known/jwks.json\n\n\nFor more information, see \n\nMCP authentication\n\nhttps://agentgateway.dev/docs/standalone/latest/configuration/security/mcp-authn/\n\n.\n\n🌟 New features\n\nThe following features were introduced in 1.0.0.\n\nSimplified LLM configuration\n\nA new top-level \n\nllm\n\n configuration section provides a simplified way to configure LLM providers. Instead of setting up the full \n\nbinds\n\n , \n\nlisteners\n\n , \n\nroutes\n\n , and \n\nbackends\n\n hierarchy, you can now define models directly in a flat structure. The simplified format defaults to port 4000.\n\nThe following example configures an OpenAI provider with a wildcard model match:\n\n# yaml-language-server: $schema=https://agentgateway.dev/schema/config\n\nllm:\n  models:\n  - name: \"*\"\n    provider: openAI\n    params:\n      apiKey: \"$OPENAI_API_KEY\"\n\n\nSetting\n\nDescription\n\nname\n\nThe model name to match in incoming requests. When a client sends \n\n\"model\": \"<name>\"\n\n , the request is routed to this provider. Use \n\n*\n\n to match any model name.\n\nprovider\n\nThe LLM provider to use, such as \n\nopenAI\n\n , \n\nanthropic\n\n , \n\nbedrock\n\n , \n\ngemini\n\n , or \n\nvertex\n\n .\n\nparams.model\n\nThe model name sent to the upstream provider. If set, this overrides the model from the request. If not set, the model from the request is passed through.\n\nparams.apiKey\n\nThe API key for authentication. You can reference environment variables using the \n\n$VAR_NAME\n\n syntax.\n\nYou can also define model aliases to decouple client-facing model names from provider-specific identifiers:\n\nllm:\n  models:\n  - name: fast\n    provider: openAI\n    params:\n      model: gpt-4o-mini\n      apiKey: \"$OPENAI_API_KEY\"\n  - name: smart\n    provider: openAI\n    params:\n      model: gpt-4o\n      apiKey: \"$OPENAI_API_KEY\"\n\n\nPolicies such as rate limiting and authentication can be set at the \n\nllm\n\n level to apply to all models:\n\nllm:\n  policies:\n    localRateLimit:\n    - maxTokens: 10\n      tokensPerFill: 1\n      fillInterval: 60s\n      type: tokens\n  models:\n  - name: \"*\"\n    provider: openAI\n    params:\n      apiKey: \"$OPENAI_API_KEY\"\n\n\nThe traditional route-based configuration ( \n\nbinds\n\n / \n\nlisteners\n\n / \n\nroutes\n\n ) remains fully supported for advanced use cases that require path-based routing or custom endpoints.\n\nFor more information, see the provider setup guides such as \n\nOpenAI\n\nhttps://agentgateway.dev/docs/standalone/latest/llm/providers/openai/\n\n, \n\nAnthropic\n\nhttps://agentgateway.dev/docs/standalone/latest/llm/providers/anthropic/\n\n, and \n\nBedrock\n\nhttps://agentgateway.dev/docs/standalone/latest/llm/providers/bedrock/\n\n.\n\nLLM request transformations\n\nYou can now use CEL expressions to dynamically compute and set fields in LLM requests. This allows you to enforce policies, such as capping token usage, without changing client code.\n\nThe following example caps \n\nmax_tokens\n\n to 10 for all requests:\n\nbinds:\n- port: 3000\n  listeners:\n  - routes:\n    - backends:\n       - ai:\n          name: openai\n          provider:\n            openAI:\n              model: gpt-3.5-turbo\n      policies:\n        backendAuth:\n          key: \"$OPENAI_API_KEY\"\n        ai:\n          transformations:\n            max_tokens: \"min(llmRequest.max_tokens, 10)\"\n\n\nFor more information, see \n\nTransform requests\n\nhttps://agentgateway.dev/docs/standalone/latest/llm/transformations/\n\n.\n\nExtended thinking and structured outputs for Claude providers\n\nExtended thinking and structured outputs are now supported for Anthropic and Amazon Bedrock Claude providers.\n\nExtended thinking\n\n lets Claude reason through complex problems before generating a response. Thinking is opt-in. You must provide specific attributes in your request to enable extended thinking.\n\nStructured outputs\n\n constrain the model to respond with a specific JSON schema. You define the JSON schema as part of your request.\n\nFor more information, see the following resources:\n\nAnthropic extended thinking and structured outputs\n\nhttps://agentgateway.dev/docs/standalone/latest/llm/providers/anthropic/\n\nBedrock extended thinking and structured outputs\n\nhttps://agentgateway.dev/docs/standalone/latest/llm/providers/bedrock/\n\nBuilt-in CEL playground\n\nThe agentgateway admin UI now includes a built-in CEL playground for testing CEL expressions against agentgateway's CEL runtime. The playground supports the custom functions and variables specific to agentgateway that are not available in generic CEL environments. To open it, select \n\nCEL Playground\n\nhttp://localhost:15000/ui/cel/\n\n in the admin UI sidebar.\n\n \n\n \n\nFor more information, see \n\nCEL playground\n\nhttps://agentgateway.dev/docs/standalone/latest/reference/cel/#cel-playground\n\n.\n\nRemote URL support for OpenAPI schemas\n\nYou can now load an OpenAPI schema from a remote URL by setting the \n\nurl\n\n field in the \n\nschema\n\n section of your OpenAPI target configuration. Agentgateway fetches the schema at startup. Previously, only local file paths and inline schemas were supported.\n\nopenapi:\n  schema:\n    url: https://example.com/api/openapi.json\n  host: example.com\n\n\nFor more information, see \n\nConnect to an OpenAPI server\n\nhttps://agentgateway.dev/docs/standalone/latest/mcp/connect/openapi/\n\n.\n\nBackend connection policies for extauth, rate limit, and MCP\n\nYou can now configure backend connection policies directly on \n\nextAuthz\n\n , \n\nremoteRateLimit\n\n , and \n\nmcp\n\n targets to control how agentgateway connects to those services. This lets you configure TLS, authentication, and connection timeouts inline, without separate policy definitions.\n\nRate limit example\n\n:\n\nremoteRateLimit:\n  host: ratelimit-service:8081\n  domain: my-api\n  policies:\n    backendAuth:\n      key:\n        file: /secrets/api-key\n    backendTLS:\n      root: /certs/ca.pem\n      insecure: false\n    tcp:\n      connectTimeout:\n        secs: 3\n        nanos: 0\n  descriptors:\n    - entries:\n        - key: service\n          value: '\"my-service\"'\n  failureMode: failOpen\n\n\nFor more information, see \n\nRate limits\n\nhttps://agentgateway.dev/docs/standalone/latest/configuration/resiliency/rate-limits/#backend-connection-policies\n\n.\n\nExtauth example\n\n:\n\nextAuthz:\n  host: authz-server:9001\n  policies:\n    backendTLS:\n      root: /certs/ca.pem\n    backendAuth:\n      key:\n        file: /secrets/api-key\n    http:\n      requestTimeout: \"5s\"\n  protocol:\n    grpc: {}\n\n\nFor more information, see \n\nExternal authorization\n\nhttps://agentgateway.dev/docs/standalone/latest/configuration/security/external-authz/#backend-connection-policies\n\nMCP example\n\n:\n\nmcp:\n  targets:\n    - name: my-mcp-server\n      sse:\n        host: mcp-backend\n        port: 8080\n      policies:\n        backendAuth:\n          key:\n            file: /secrets/mcp-token\n        backendTLS:\n          root: /certs/ca.pem\n\n\nRemote rate limit failure modes\n\nYou can now configure how agentgateway behaves when the remote rate limit service is unavailable using the new \n\nfailureMode\n\n field. The default behavior is \n\nfailClosed\n\n , which denies requests with a \n\n500\n\n status code. Set \n\nfailureMode\n\n to \n\nfailOpen\n\n to allow requests through when the service is unreachable.\n\nremoteRateLimit:\n  host: localhost:9090\n  domain: example.com\n  failureMode: failOpen\n  descriptors:\n  - entries:\n    - key: organization\n      value: 'request.headers[\"x-organization\"]'\n    type: requests\n\n\nFor more information, see \n\nFailure behavior\n\nhttps://agentgateway.dev/docs/standalone/latest/configuration/resiliency/rate-limits/#failure-behavior\n\n.\n\nJWT claim validation for MCP auth\n\nYou can now customize which JWT claims must be present in a token before it is accepted, using the new \n\njwtValidationOptions.requiredClaims\n\n field in your MCP authentication configuration.\n\nmcpAuthentication:\n  issuer: http://localhost:9000\n  jwks:\n    url: http://localhost:9000/.well-known/jwks.json\n  jwtValidationOptions:\n    requiredClaims:\n      - exp\n      - aud\n      - sub\n\n\nFor more information, see \n\nJWT claim validation\n\nhttps://agentgateway.dev/docs/standalone/latest/mcp/mcp-authn/#jwt-claim-validation\n\n.\n\n🪲 Bug fixes\n\nMCP per-request policy evaluation\n\nMCP policies are now re-evaluated on each request rather than only at session start. If an operator updates an authorization policy, such as by revoking access to a tool or changing JWT claim requirements, the change takes effect immediately on the next request, without requiring the client to tear down and re-establish the MCP session.\n\nNote that this is a behavioral improvement. Existing MCP authorization configuration benefits automatically.\n\nCORS evaluation ordering\n\nCORS evaluation now runs \n\nbefore\n\n authentication and \n\nbefore\n\n rate limiting. Previously, CORS ran after auth and rate limiting, which caused two problems:\n\nBrowser preflight OPTIONS requests were rejected by auth, making cross-origin requests impossible when auth was enabled\n\nRate-limited 429 responses lacked CORS headers, so browsers saw an opaque CORS error instead of a retryable one\n\nNote that this is a behavioral improvement. Existing configurations that combine CORS policies with extauth and rate limiting policies now work correctly.\n\nConfiguration reference\n\nhttps://agentgateway.dev/docs/standalone/latest/reference/configuration/\n\nAsk AI\n\nAgentgateway assistant\n\nAsk me anything about agentgateway configuration, features, or usage.\n\nNote: AI-generated content might contain errors; please verify and test all returned information.\n\nTip: one topic per conversation gives the best results.
