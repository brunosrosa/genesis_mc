None of them block one another: the model continues serving requests while training runs in the background, and judging happens concurrently with new interactions.\n\nSelf-Hosted & Private by Design\n\nThe entire stack, including the \n\npolicy model\n\n, \n\njudge/PRM\n\n, and \n\ntrainer\n\n, runs on \n\nyour own infrastructure\n\n. Conversation data stays within your system, and no third-party model API is required.\n\nFrom Feedback to Gradient — Automatically\n\nYou do not need to manually label data. The system automatically:\n\nOrganizes multi-turn interactions into session-aware training trajectories\n\nClassifies API messages into \n\nmain-line\n\n (trainable) vs. \n\nside\n\n (non-trainable) turns\n\nUses the next user, environment, or tool feedback as a natural \"next-state\" signal\n\nRuns PRM/judge evaluation asynchronously, with majority voting when needed for more robust scoring\n\nSubmits ready samples to the trainer as they become available\n\nThree Optimization Methods in One Framework\n\nBinary RL (GRPO):\n\n A Process Reward Model scores each turn based on next-state feedback. The scalar reward is then used with GRPO advantage estimation and a PPO-style clipped surrogate loss.\n\nOn-Policy Distillation (OPD):\n\n When the next state reveals useful hindsight, a judge model extracts a textual hint. This hint augments the original prompt to create an enhanced teacher, whose token-level log-probability gap with the student becomes a directional advantage signal richer than any scalar reward.\n\nCombination Method:\n\n OpenClaw-RL further combines Binary RL and OPD in a unified training recipe, leveraging the dense scalar supervision of Binary RL together with the richer token-level directional signal from OPD.
