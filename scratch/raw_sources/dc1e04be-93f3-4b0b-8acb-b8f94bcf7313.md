One coordinator agent handles user interaction, memory, and routing. A cloud agent burns through $500 of Opus tokens chasing a hallucinated Terraform state or decides to reconfigure your Istio ambient mesh routing because it misread a waypoint proxy status. A general agent with SSH access starts “fixing” things on production hosts that don't need fixing.\n\nWithout a control plane between your agents and the outside world, you have no way to stop any of this. No kill switch. No cost ceiling. No audit trail. No rate limits. Just agents with direct access to LLMs and tools, hoping nothing goes wrong.\n\nThat's not engineering. That's negligence.\n\nThe Architecture\n\nHere's what I actually run. It owns the conversation, manages memory (a persistent \n\nMEMORY.md\n\n that carries context across sessions), and decides which specialist to invoke for each request.\n\nWhen a task comes in, the coordinator classifies it and builds a context payload — the relevant portion of memory, the specific question, any constraints — and spawns a specialist. The specialist does its work, returns a result, and dies. Stateless. Disposable.\n\nThe coordinator synthesizes the result and delivers it back to the user. If a task spans multiple domains, the coordinator fans out to multiple specialists in parallel.\n\nModel\n\n: Sonnet — fast enough for routing, smart enough for context assembly.\n\nSecurity search.\n\nTwo memory layers — Daily logs (memory/YYYY-MM-DD.md) for running notes, and MEMORY.md for curated long-term facts and preferences.\n\nDisk-based memory — No RAM between sessions, so everything must be written to disk to be remembered.\n\nVector embeddings — Files are chunked and converted to vectors using Qwen3-Embedding-0.6B via node-llama-cpp, fully local and private.\n\nHybrid search — Combines vector similarity (semantic) with BM25 full-text search (keyword) for accurate recall.\n\nTemporal decay — Recent memories are weighted higher in search results.\n\nmemory_search / memory_get — Agent tools to find relevant past context or read a file directly.\n\nAuto-compaction — When token limits are hit, the agent is prompted to save important context before compacting so Here's why I don't let any agent — not even the coordinator — talk to LLMs or tools directly.\n\nThe Doom Scenario\n\nPicture this: your cloud agent is debugging a Terraform plan. It calls Opus to reason about a complex state migration. The model hallucinates a resource dependency. The agent re-plans, calls the model again for clarification, gets another hallucination, retries with more context (bigger prompt, more tokens), and enters a loop. Each iteration costs more than the last because the context window keeps growing.\n\nWithout a gateway: you find out when the invoice arrives. $2,000 spent on a conversation with itself.\n\nWith agentgateway: the agent hits a token-per-minute ceiling after the third iteration. The request is rejected. You get an alert. You investigate. Total damage: $12.\n\nThat's not a hypothetical. That's Tuesday.\n\nKill Switch\n\nagentgateway gives me a single point where I can shut everything down. If I see an agent misbehaving — through the metrics, through the traces, through an alert — I can:\n\nRevoke the JWT\n\n for that specific agent's identity. Immediate. That agent can't make another LLM call or tool invocation.\n\nUpdate the rate limit\n\n to zero for that agent class. Every security agent stops. Every cloud agent stops. Surgical.\n\nPull the gateway entirely.\n\n Nuclear option. Everything stops. Nothing reaches any LLM or tool.\n\nWithout a gateway, killing a rogue agent means finding the pod, kubectl exec-ing into the right node, and hoping you're faster than the agent. With a gateway running in Kubernetes, it's a config change — or a \n\nkubectl rollout restart\n\n away from a full reset.\n\nCost Controls\n\nEvery agent has a budget. Not a suggestion — a hard limit enforced at the gateway level.\n\n# Security agent route — Opus workloads\npolicies:\n  localRateLimit:\n    - maxTokens: 50000\n      tokensPerFill: 50000\n      fillInterval: 1m\n      type: tokens\n    - maxTokens: 20\n      tokensPerFill: 20\n      fillInterval: 1m\n      type: requests\n\n# Cloud agent route — higher throughput\npolicies:\n  localRateLimit:\n    - maxTokens: 100000\n      tokensPerFill: 100000\n      fillInterval: 1m\n      type: tokens\n    - maxTokens: 30\n      tokensPerFill: 30\n      fillInterval: 1m\n      type: requests\n\n# General agent route — simple ops\npolicies:\n  localRateLimit:\n    - maxTokens: 20000\n      tokensPerFill: 20000\n      fillInterval: 1m\n      type: tokens\n    - maxTokens: 15\n      tokensPerFill: 15\n      fillInterval: 1m\n      type: requests\n\n\nEach route gets a token-bucket rate limit scoped by the route's identity. The security agent running Opus gets 50k tokens per minute. That's enough for serious threat analysis but not enough to bankrupt me on a hallucination loop. The general agent on Haiku gets 20k — simple ops don't need more.\n\nagentgateway tracks token usage per provider and per model with \n\nagentgateway_gen_ai_client_token_usage\n\n metrics, tagged with provider, model, and operation labels. I know exactly what each agent costs, in real time.\n\nRate Limiting\n\nRate limits aren't just about cost. They're about preventing an agent from overwhelming a downstream system.\n\nA network agent running \n\nnmap\n\n scans through an MCP tool server could, in theory, scan your entire /16 network if nobody stops it. Rate limiting at the gateway means the agent gets N tool calls per minute, period. It can't outrun the limit no matter how convinced it is that it needs to scan “just one more subnet.”\n\nSame for LLM calls. An agent that retries on every 429 or timeout — something LLM providers actually rate-limit you for — gets its retries throttled at the gateway before the provider even sees them.\n\nGovernance and RBAC\n\nEach agent has a JWT identity with scoped permissions. The security agent can call \n\nnmap\n\n and \n\ntrivy\n\n tools but cannot call \n\nterraform apply\n\n . The cloud agent can call \n\nterraform plan\n\n but not \n\nssh\n\n . The general agent can SSH to designated hosts but cannot touch cloud credentials.\n\nThis is enforced at the gateway with CEL expressions in \n\nmcpAuthorization\n\n rules:\n\n# Security agent backend\nmcpAuthorization:\n  rules:\n  - >-\n    jwt.agent_role == \"security\" && (\n      mcp.tool.name.startsWith(\"nmap\") ||\n      mcp.tool.name.startsWith(\"trivy\") ||\n      mcp.tool.name.startsWith(\"falco\")\n    )\n\n# Cloud agent backend\nmcpAuthorization:\n  rules:\n  - >-\n    jwt.agent_role == \"cloud\" && (\n      mcp.tool.name.startsWith(\"terraform\") ||\n      mcp.tool.name.startsWith(\"kubectl\") ||\n      mcp.tool.name.startsWith(\"istioctl\") ||\n      mcp.tool.name.startsWith(\"helm\")\n    )\n\n\nEven if a specialist agent's system prompt gets jailbroken and it tries to invoke tools outside its domain, the gateway blocks it. If a tool isn't matched by a rule, it's automatically filtered from the \n\ntools/list\n\n response — the agent literally cannot see tools it doesn't have access to.\n\nAnd since unmatched tools are denied by default, I only whitelist what's explicitly allowed. Any tool not covered by an \n\nmcpAuthorization\n\n rule is invisible to the agent. For HTTP-level operations, I add explicit deny rules:\n\nauthorization:\n  rules:\n  - deny: 'request.path.contains(\"delete\")'\n  - deny: 'request.path.contains(\"destroy\")'\n  - deny: 'request.path.contains(\"drop\")'\n\n\nNo agent gets to run destructive operations without explicit human escalation. Period.\n\nFull Observability\n\nEvery LLM call and every tool invocation generates OpenTelemetry traces. No long-running agent processes consuming resources while idle. The coordinator is the only persistent component.\n\nCoordinator owns all memory.\n\n Specialists get context injected per request. They don't need to remember previous conversations — the coordinator handles continuity.\n\nModel per agent.\n\n Opus for security (high-stakes reasoning). Sonnet for network/cloud (speed + capability balance). Haiku for simple ops (cost efficiency). Each agent gets the cheapest model that's good enough for its domain.\n\nTool isolation.\n\n Each specialist only gets the tools it needs. Not through prompt instructions (which can be jailbroken) but through gateway-enforced RBAC (which can't).\n\nSingle gateway for all traffic.\n\n Not one gateway per agent. Not a sidecar pattern. One agentgateway instance running in Kubernetes that every agent routes through. One place to set policy, one place to monitor, one place to kill. K8s gives me rolling updates, health checks, and resource limits on the gateway itself — so the control plane has its own control plane.\n\nExtensible.\n\n New domain = new agent config + system prompt + tool set. The coordinator's routing logic gets a new keyword match. The gateway gets a new JWT scope. No architectural changes needed.\n\nRunning agentgateway in Kubernetes\n\nagentgateway runs as a deployment in my Kubernetes cluster. This isn't just convenience — it's operational discipline. The gateway that controls all my agents is itself managed by K8s primitives: health checks, resource limits, rolling updates, and restart policies.\n\nThe kill switch becomes even simpler in K8s. Scale to zero replicas and every agent loses its gateway instantly:\n\nkubectl scale deployment agentgateway -n agent-infra --replicas=0\n\n\nEverything stops. Scale back up when you've fixed the issue. The gateway comes back with the same config, same policies, same state.\n\nThe cloud agent — the one that handles Kubernetes, Istio, and ambient mesh — is particularly interesting in this setup. It manages the same cluster that hosts the gateway. That's a circular dependency I've thought carefully about: the agent that manages K8s infrastructure talks through a gateway that runs on K8s infrastructure. The circuit breaker here is the RBAC policy — the cloud agent's JWT scope explicitly excludes the \n\nagent-infra\n\n namespace. It can manage workloads, configure Istio routing, and deploy ambient mesh policies, but it cannot touch the gateway deployment itself.\n\nWhy agentgateway and Not a Traditional Proxy\n\nTraditional API gateways (Envoy, Kong, NGINX) were built for HTTP request/response. AI agent traffic is fundamentally different:\n\nMCP is stateful.\n\n Agents maintain long-lived sessions with tool servers. Requests and responses are tied to session context. Traditional gateways don't maintain session awareness.\n\nLLM calls are long-running.\n\n A single inference call can take 30+ seconds with streaming. Connection timeouts designed for web APIs don't apply.\n\nToken-based economics.\n\n Cost isn't about request count — it's about token count. A gateway that can't count tokens can't enforce budgets.\n\nBidirectional communication.\n\n MCP servers can push messages back to clients asynchronously. This breaks the request/response model traditional gateways assume.\n\nagentgateway\n\nhttps://agentgateway.dev/\n\n is purpose-built for this. Written in Rust for performance and memory safety on stateful, long-lived connections. Understands MCP sessions natively. Counts tokens per-provider. Handles fan-out patterns where one agent call becomes multiple downstream requests.\n\nIt's open source, Apache 2.0 licensed, and part of the Linux Foundation. No vendor lock-in.\n\nThe Takeaway\n\nA multi-agent system without a control plane is a liability. Every agent you deploy is a potential cost bomb, a potential security breach, a potential “I can't believe nobody caught that” incident.\n\nThe architecture is straightforward:\n\nOne coordinator\n\n that handles users and routes tasks\n\nSpecialist agents\n\n that are stateless, scoped, and disposable\n\nOne gateway\n\n that sees everything, controls everything, and logs everything\n\nThe coordinator decides \n\nwhat\n\n gets done. The gateway decides \n\nwhether\n\n it's allowed to happen. That separation is what makes the system safe to run autonomously.\n\nagentgateway isn't optional in this architecture. It's the thing that makes the entire system possible without me staring at a terminal 24/7 wondering if an agent is about to do something catastrophic.\n\nBuild the agents. Put the gateway in front.
